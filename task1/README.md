## Описание алгоритма

### Задача
Подсчёт количества уникальных IPv6-адресов в текстовом файле объёмом до 10^9 строк при ограничении оперативной памяти в 1 ГБ.

### Нормализация адресов
Каждый IPv6-адрес приводится к каноническому бинарному представлению -- 16 байт в формате big-endian. Раскрывается `::`, удаляются ведущие нули, регистр приводится к нижнему. Используется `struct.pack(">8H", ...)`. Лексикографический порядок полученных байтов совпадает с числовым порядком адресов.

### Базовое решение (критерий III)
Все адреса хранятся в `set` в оперативной памяти. Файл читается построчно, каждый адрес нормализуется и добавляется в множество. Результат — `len(seen)`. Работает корректно при достаточном объёме RAM.

### Оптимизированное решение (критерий IV)
Реализован алгоритм внешней сортировки (external merge sort) с подсчётом уникальных на фазе слияния.

**Фаза 1 -- разбиение и сортировка:**
Файл читается частями фиксированного размера (chunk). Размер чанка рассчитывается исходя из бюджета памяти 800 МБ, с учётом накладных расходов Python на объект (~57 байт на запись: 16 байт данных + 33 байта заголовок PyObject + 8 байт указатель в списке). Каждый чанк сортируется in-place и записывается в бинарный временный файл (16 байт на запись). Запись ведётся поэлементно с буфером 1 МБ, что исключает пиковое выделение памяти на `b"".join()`. После записи вызывается `gc.collect()` для возврата памяти ОС.

**Фаза 2 -- k-way merge:**
Все временные файлы сливаются через `heapq.merge` (min-heap, O(log k) на элемент, реализация на C). Из каждого файла записи читаются генератором с буферизацией. При слиянии поддерживается переменная `prev` — предыдущее значение. Если текущее значение отличается от `prev`, счётчик уникальных увеличивается на 1. По завершении все временные файлы удаляются.

### Параллельная версия (критерий V)
Фаза 1 распараллелена через `multiprocessing.Pool`. Входной файл делится на сегменты по байтовым смещениям с выравниванием по границам строк. Каждый воркер открывает файл самостоятельно и делает `seek` к своему сегменту — через IPC передаются только пути к временным файлам, данные не сериализуются.

Количество воркеров ограничено двумя факторами: числом доступных ядер CPU и бюджетом памяти. Вычисляется `min(cpu_count - 1, memory_budget / (min_chunk × bytes_per_record))`. Это гарантирует, что суммарное потребление памяти всех воркеров не превышает 800 МБ при любом количестве ядер.

Фаза 2 выполняется последовательно -- она ограничена скоростью дискового ввода-вывода.

### Структура репозитория
- `task1_basic.py` -- базовое решение (set в памяти)
- `task1.py` -- оптимизированное параллельное решение

Используются только модули стандартной библиотеки Python: `struct`, `heapq`, `tempfile`, `multiprocessing`, `gc`.

### Запуск
Базовое решение:
```bash
python task1_basic.py input.txt output.txt
```

Оптимизированное решение:
```bash
python task1.py input.txt output.txt
```
